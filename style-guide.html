Overview

Design Principles

Seven principles that govern decisions the system doesn't explicitly answer. When a choice isn't covered by a rule, apply the nearest principle.

01
Clarity over density
Show less when in doubt. Every additional element competes for attention. The user's cognitive load is the real constraint.
02
Consistency over novelty
Use an existing pattern before inventing a new one. A familiar interaction is faster than a clever one.
03
Accessibility by default
WCAG AA is a floor, not a goal. Build for keyboard, screen reader, and high contrast from the start.
04
One obvious action per context
Every view has one primary action. If two feel equally important, one of them isn't the primary yet.
05
Progressive disclosure
Show what's needed now. Reveal complexity on demand. Never make the simple case harder to serve the advanced case.
06
Every action has feedback
The system always responds — loading, success, or error. Silent failures are not acceptable.
07
Prevent errors before recovering from them
Constrain inputs, confirm destructive actions, validate inline. Recovery is always more expensive than prevention.

Typography Roles

Six semantic roles for every text context. Use role classes (.t-display, .t-title, etc.) in all new markup — never hardcode sizes. Size utilities are kept as implementation internals only.

Role scale

.t-body Default body text and form labels 14px · 400 · Prose, form labels

.t-label Badge · Metadata · Timestamp 12px · 500 · Labels, badges
.t-caption Helper text and footnotes 11px · 400 · Captions, hints
Form label rule
✓ Correct — .t-body (14px)
✗ Wrong — .t-label (12px)

Enterprise users spend their working lives in forms. 14px labels improve scannability and pass WCAG 1.4.4 at 200% zoom.

Foundations

Z-Index Layers

Named semantic layers. Never use raw z-index values — always reference a layer variable. Tooltip always wins at 700, sitting above modals (500) so a tooltip inside a modal renders correctly.

700Tooltip$z-tooltipAlways highest — must appear above modals
600Toast$z-toastToast notifications
500Modal$z-modalModals, dialogs
400Overlay$z-overlayModal backdrop / scrim
300Popover$z-popoverInteractive overlays (click-triggered, focus-managed)
200Dropdown$z-dropdownDropdown menus, contextual menus
100Sticky$z-stickySticky headers, fixed top nav
0Base$z-baseDefault document flow

Popover vs Tooltip: Popovers are click-triggered, interactive, and focus-managed (role="dialog"). Tooltips are hover-triggered, non-interactive, and never contain focusable elements (role="tooltip").

Theming

Keyboard Standards

Every interactive component must be fully operable by keyboard. Tab moves forward, Shift+Tab backward, Esc closes overlays. Focus is never trapped except inside an open modal.

Modal / Dialog
KeyBehaviour
TabCycles forward through focusable elements inside the modal only
Shift+TabCycles backward — never leaves the modal
EscCloses. Focus returns to the element that triggered it
EnterActivates the focused button

On open: focus moves to first focusable element. Background is inert while modal is open.

Dropdown / Contextual Menu
KeyBehaviour
Enter / Space on triggerOpens dropdown. Focus moves to first item
↓ / ↑Moves focus between items. Wraps at edges
Home / EndJumps to first / last item
Enter / Space on itemSelects and closes
EscCloses. Focus returns to trigger
Tabs / Segmented Control
KeyBehaviour
TabMoves focus into the tab group (to active tab), then out
→ / ←Moves focus to next/previous tab and activates it immediately
Home / EndFirst / last tab

Uses roving tabindex — only the active tab is in the tab order. Arrow keys navigate, not Tab.

Form controls — Select
KeyBehaviour
Space / ↓Opens the select
↓ / ↑Navigates options
EnterSelects focused option and closes
EscCloses without changing selection
Data Table
KeyBehaviour
TabMoves between interactive elements: checkboxes, headers, row actions
Space on checkboxToggles row selection
Enter on headerToggles sort direction
Enter on actionActivates that row action

Dark Mode

Token architecture is fully prepared. Visual implementation is scheduled for v2. No component API changes required — only token values switch.

Coming in v2

Dark mode tokens are ready

All component styles use CSS custom properties. Add the .dark class to <html> to activate — zero component changes needed. This aligns with Tailwind's darkMode: "class" strategy and shadcn/ui convention.

/* One-line toggle */
document.documentElement.classList.add('dark');

.dark {
  --surface-page:   #0F172A;  --surface-base:   #1E293B;
  --text-primary:   #F1F5F9;  --border-default: #334155;
  /* ... all tokens defined in DESIGN_SYSTEM.md */
}

QA Checklists

Two separate checklists for two audiences. Design review asks "Does this feel right?" Engineering review asks "Can this ship safely?" Never combine them.

Design Review
Audience: designers, Claude output review  ·  "Does it feel like IMIN?"
Visual
Spacing uses system tokens — no hardcoded pixel values
Typography uses semantic role classes (.t-display, .t-title, etc.)
Exactly one .btn--primary per view — never two
Alignment is consistent — elements share axis lines
Truncation rules applied — long text ellipses correctly
States
Loading state shown — skeleton matches the loaded layout
Empty state shown — title + description + action
Error state shown — has a retry path
All interactive states visible: hover, focus, active, disabled
Content
Error messages: [what went wrong] + [how to fix it]
Confirmation titles state consequence, not action
Toast copy: past tense for success, present for error
Theme
Reviewed in both light and dark mode
No hardcoded colour values visible in dark mode
Engineering QA
Audience: engineers, PR review, AI output  ·  "Can it ship safely?"
Accessibility
All elements reachable by keyboard (Tab / Shift+Tab)
Focus ring visible on every focused element
Contrast AA verified: text ≥ 4.5:1, large text ≥ 3:1
Form inputs have associated <label> elements
ARIA roles and states match Section 13 keyboard specs
Modal: focus trapped, background inert, returns to trigger
Functionality
Validation fires on blur — not on keystroke
Disabled states use disabled attr or aria-disabled — not CSS only
Loading states use aria-busy="true"
All error states are recoverable
Responsive
Tested at 1440px, 1280px, 1024px, 768px
Tables and Kanban scroll horizontally at narrow widths
Touch targets ≥ 44×44px
Dark mode & performance
No hardcoded hex values in component CSS
Z-index uses named layer variables ($z-modal, $z-tooltip, etc.)
prefers-reduced-motion respected
Animations use transform and opacity only
